## 회원가입 (Sign In) 흐름
회원가입과 로그인을 하나의 API로 통합  
기존 유저가 없으면 자동으로 생성(가입)하고, 있으면 그대로 로그인

### 전체 시퀀스
Client                                                                          
│
│  ( 클라이언트가 서버에 요청 )
│  POST /api/sign/v1/in/google                                                
│  Body: { "accessToken": "<Google OAuth2 토큰>" }                            
│                                                                             
▼                                                                             
SignApi (@ApiRestController)            
│  
│  provider 문자열 → UserCredentialsProvider.of("google") → GOOGLE enum 변환  
│
▼
SignApplication (@Service)  ← 핵심 오케스트레이션
│
│  ① provider가 GOOGLE인지 검증
│    ( GOOGLE이 아니면 예외 )
│
│  ② UserSignService.googleOAuth2(accessToken)
│     │
│     │ ( 이 안에서 하는 일들 )
│     ▼
│     1. GoogleOAuth2Client (OpenFeign) → 구글 API 호출
│     │  GET https://www.googleapis.com/oauth2/v2/userinfo
│     │  Header: Authorization: Bearer {accessToken}
│     │  ( accessToken 통해 사용자 정보 받기 )
│     │
│     ▼
│     2. GoogleOAuth2UserInfo 응답 수신
│     │  - id, email, name, picture, verifiedEmail ...
│     │    ( id는 구글이 주는 유저의 고유 ID )
│     │ 
│     ▼
│     3. 이메일 인증 여부 확인 (verifiedEmail == false → 예외)
│     │
│     ▼
│     SignInfo(email, providerId, provider) 반환
│
│  ③ 기존 유저 조회 or 신규 생성
│     userRepository.findByEmail(email)
│       ├── 있으면 → 기존 User 사용 (로그인)
│       └── 없으면 → User.new(email)  (회원가입)
│                     ├── id: ULID 생성
│                     ├── status: JOINER
│                     └── createdAt: UTC now
│
│  ④ 기존 Credentials 조회 or 신규 생성
│     userCredentialsRepository.findByProviderAndProviderId(GOOGLE,
│                                                   providerId)
│       ├── 있으면 → 기존 UserCredentials 사용
│       └── 없으면 → UserCredentials.new(userId, provider, providerId)
│
│  ⑤ User & UserCredentials 저장 (DB)
│     userRepository.save(user)
│     userCredentialsRepository.save(userCredentials)
│
│  ⑥ JWT 토큰 발급
│     jwtTokenProvider.generate(userId)
│       ├── subject: userId
│       ├── expiration: 30일
│       └── algorithm: HS512
│
▼
SignTokenView { accessToken: "<JWT>" }  →  Client에 반환


### 핵심 포인트
1. 가입/로그인 통합 — email 기준으로 findByEmail 후 없으면 User.new()로 자동
   가입
2. Google OAuth2 전용 — 현재 GOOGLE provider만 허용 (FACEBOOK은 enum에만 존재)
3. 초기 상태 — 가입 시 UserStatus.JOINER로 시작
4. 토큰 — JWT (HS512, 30일 만료), subject에 userId 저장

### 공통상수
**UserCredentialsProvider**
OAuth2 소셜 로그인 제공자를 구분하는 enum
```
  enum class UserCredentialsProvider {
      GOOGLE,
      FACEBOOK;
  }
```
- 현재 GOOGLE, FACEBOOK 두 값이 정의되어 있지만, 실제 로직에서는 GOOGLE만
  허용합니다.
- provider != GOOGLE이면 예외 발생
- DB에는 @Enumerated(EnumType.STRING)으로 "GOOGLE" 문자열 그대로 저장됩니다.


**UserCredentials**
User와 외부 OAuth2 Provider 간의 연결 정보를 담는 도메인 모델
User와 분리되어 있어서 한 명의 유저가 여러 소셜 계정을 연결할 수 있는 구조
```
 필드 구성

  필드: id
  타입: String
  설명: PK, ULID 자동 생성
  ────────────────────────────────────────
  필드: userId
  타입: UserId (Integer)
  설명: users 테이블의 FK — 어떤 유저의 인증 정보인지
  ────────────────────────────────────────
  필드: providerId
  타입: String
  설명: OAuth2 provider가 발급한 고유 ID (Google의 사용자 ID)
  ────────────────────────────────────────
  필드: provider
  타입: UserCredentialsProvider
  설명: 어떤 소셜 로그인인지 (GOOGLE, FACEBOOK)
  ────────────────────────────────────────
  필드: createdAt
  타입: LocalDateTime
  설명: 생성 시각 (UTC)

```

### DB 테이블

users_credentials  
├── id           (PK, VARCHAR)  
├── user_id      (FK → users.id)  
├── provider_id  (VARCHAR, Google 사용자 고유 ID)  
├── provider     (VARCHAR, "GOOGLE")  
└── created_at   (TIMESTAMP)  


### User와의 관계

users (1) ──── (*) users_credentials  
│                    │  
│ id ◄───────── user_id  
│                    │  
│                provider = GOOGLE  
│                providerId = "1234567890" (Google 사용자 ID)  


한 User가 여러 UserCredentials를 가질 수 있는 1:N 관계입니다. 예를 들어 같은
유저가 Google과 Facebook 모두 연결하면 users_credentials 레코드가 2개 생깁니다.


### 회원가입 시 동작 흐름

1. Google OAuth2 API로부터 providerId(Google 사용자 ID)를 받음  

2. findByProviderAndProviderId(GOOGLE, providerId)로 기존 Credentials 조회  
   ├── 있으면 → 기존 것 재사용 (이미 가입된 유저)  
   └── 없으면 → UserCredentials.new() 생성 (신규 가입)  
   ├── id: PK 생성  
   ├── userId: User의 id 연결  
   ├── provider: GOOGLE  
   ├── providerId: Google 사용자 ID  
   └── createdAt: UTC now  

3. userCredentialsRepository.save()로 DB 저장

### 핵심 정리
- Provider — "어떤 소셜 로그인을 사용했는가" (Google? Facebook?)
- UserCredentials — "이 유저가 이 소셜 계정으로 가입했다"는 연결 기록
- 이 분리 덕분에 나중에 Apple, Kakao 등 새 provider를 추가할 때
  UserCredentialsProvider enum에 값을 추가하고, 해당 OAuth2 Client만 구현하면
  됩니다. User 도메인은 변경할 필요가 없습니다.
